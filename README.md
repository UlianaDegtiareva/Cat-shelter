# 🐾 Система управления приютом для кошек (Cat Shelter API)

## Постановка задачи №5: Ролевая модель и медицинский учет

**Цель**: Расширить функционал системы за счет внедрения иерархии прав доступа (RBAC) и углубленного учета состояния здоровья животных. Теперь система должна не только хранить анкеты кошек, но и отслеживать их ветеринарную историю, при этом критические операции (такие как удаление данных или назначение прав) должны быть строго разграничены между обычными сотрудниками и администраторами.

**Основные задачи**:

1. Реализовать RolesGuard для проверки уровня доступа (Admin vs User/Volunteer).
2. Внедрить сущность HealthCard (связь One-to-One) для ведения медицинских записей.
3. Обеспечить безопасность персональных данных (возможность удаления профиля только владельцем или администратором).

### Модуль управления доступом и здоровьем

| Метод | Путь | Описание | Правила доступа (Business Logic) |
| :--- | :--- | :--- | :--- |
| **POST** | `/users/:id/make-admin` | Повышение прав | Только Admin. Позволяет назначить другому пользователю права администратора. |
| **DELETE** | `/users/:id` | Удаление профиля | Admin (любого) или Владелец (только свой ID). Защита от несанкционированного удаления чужих аккаунтов. |
| **POST** | `/cats/:id/health-card` | Создание медкарты | Любой авторизованный сотрудник. Инициализирует запись о здоровье. У одной кошки может быть только одна карта. |
| **PATCH** | `/cats/:id/health-card` | Обновление медкарты | Любой авторизованный сотрудник. Позволяет вносить правки в статус здоровья или даты прививок. |

### Ожидаемые форматы ответов (API Contract)

Ниже представлены статусы ответов, которые система должна возвращать при обращении к эндпоинтам. Убедитесь, что описания в коде и декораторах @ApiResponse совпадают с указанными ниже.
**Модуль медицинского учета (Health Module):**

1. Создание медкарты (`POST /cats/:id/health-card`)
   * **201 Created**: "Health card created successfully."
   * **404 Not Found**: "Cat not found."
   * **409 Conflict**: "This cat already has a health card."
2. Обновление медкарты (`PATCH /cats/:id/health-card`)
   * **200 OK**: "Health card updated successfully."
   * **404 Not Found**: "Health card not found for this cat"

При попытке удалить запись о пользователе, при этом не являясь ее хозяином или админом, необходимо предусмотреть предупреждение:
`@ApiResponse({ status: 403, description: 'Access denied. You can only delete your own profile'})`

Для всех методов, которые требуют авторизации перед выполнением необходимо прописать:
`@ApiResponse({ status: 401, description: 'Not authorized: No token provided or token invalid.' })`

### Инициализация системы

Для обеспечения работоспособности системы при первом запуске необходимо реализовать механизм автоматического заполнения базы данных.

Система должна автоматически создавать базовые роли в таблице roles, если они отсутствуют:

* volunteer (ID: 1) — базовая роль для всех регистрирующихся пользователей.
* admin (ID: 2) — роль с расширенными правами управления.

Необходимо предусмотреть автоматическое создание «главного администратора» при инициализации приложения.
**Параметры учетной записи:**

* Логин (Username): superadmin
* Пароль: 123456 (при сохранении в БД должен быть обязательно хеширован с использованием bcrypt).
* Имя/Фамилия: Admin Glavniy
* Роль: admin (ID: 2)

## Теоретическая часть для выполнения задания

1. **Связь One-to-One (Один-к-одному)**
   В реляционных базах данных это связь, где одной записи в таблице `A` соответствует ровно одна запись в таблице `B`.
   * В нашем случае: `CatEntity <-> HealthCard`. Это позволяет вынести медицинские подробности в отдельную таблицу, не загромождая основную анкету животного.
2. **RBAC (Role-Based Access Control)**
   Метод управления доступом, при котором права распределяются на основе ролей пользователей.
   * `Roles Guard`: Специальный механизм в NestJS, который считывает метаданные из декоратора `@Roles()` и сравнивает роль пользователя из его JWT-токена с требуемой для выполнения запроса.
3. **Иерархическая логика удаления**
   При реализации удаления пользователя применяется комбинированная проверка:
   * Проверка по ID: `req.user.id === targetId` (Владелец).
   * Проверка по роли: `req.user.role === 'admin'` (Администратор). Если ни одно условие не выполнено, система выбрасывает `ForbiddenException`.

   * Перейдите в [онлайн-редактор](https://editor.swagger.io/).

## Инструкция по запуску

### 1. Первый запуск (Настройка)

Выполняется один раз при установке проекта.

1.  **Установка зависимостей:**
    ```bash
    npm install
    ```
2.  **Запуск базы данных (Docker):**
    Убедитесь, что Docker Desktop запущен, затем выполните:
    ```bash
    docker-compose up -d
    ```
    *Система создаст контейнер с PostgreSQL на порту `5433`.*

3.  **Запуск приложения:**
    ```bash
    npm run start:dev
    ```
    При возникновении ошибок на данном шаге типа:
    `EADDRINUSE: address already in use :::3000` необходимо сменить порт в самом коде или остановить процессы, которые занимают данный порт, так как ошибка означает, что порт 3000 уже занят. Освободить порт можно при помощи команды:
    ```bash
    Stop-Process -Id (Get-NetTCPConnection -LocalPort 3000).OwningProcess -Force
    ```
    После этого снова выполните запуск приложения `npm run start:dev`

### 2. Последующие запуски

Для повседневной работы достаточно убедиться, что контейнер БД запущен, и запустить сервер:
1. `docker-compose start` (если контейнер был остановлен)
2. Для этого задания потребуются новые библиотеки, установить которые можно при помощи команды:
   ```bash
      npm install @nestjs/jwt @nestjs/passport bcrypt passport-jwt
   ```
3. `npm run start:dev`

Приложение будет доступно по адресу: [http://localhost:3000/api](http://localhost:3000/api)
