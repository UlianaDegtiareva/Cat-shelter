# 🐾 Система управления приютом для кошек (Cat Shelter API)

## Постановка задачи №3: Аналитический модуль

**Цель**: Накопленных данных стало достаточно для глубокого анализа. Теперь важно реализовать модуль отчетности, который поможет администратору приюта оценивать эффективность работы и понимать структуру "фонда" животных. Необходимо продемонстрировать умение работать со сложными SQL-запросами через **TypeORM QueryBuilder**, агрегацией данных и объединением таблиц.

**Основная задача** - создать новый модуль с эндпоинтами по пути `/stats`:

### Модуль статистики (`StatsModule`)

| Метод | Путь | Описание | Бизнес-логика |
| :--- | :--- | :--- | :--- |
| **GET** | `/stats/summary` | Общие метрики | Общее кол-во животных, кол-во усыновленных и % (Adoption Rate). |
| **GET** | `/stats/breeds` | Анализ пород | Группировка по породам с подсчетом количества в каждой. |
| **GET** | `/stats/top-adopters` | Рейтинг владельцев | Топ-5 пользователей, усыновивших наибольшее количество кошек. |

Также необходимо обновить метод `GET /cats`, добавив поддержку **сложной фильтрации**:
* Добавить необязательный параметр `isKitten` (boolean).
* Если параметр `true`, система должна возвращать только животных в возрасте до 1 года.
* Фильтр должен корректно комбинироваться с другими параметрами (`breed`, `isAdopted`).

### Ожидаемые форматы ответов

1. Пример отчета по породам
   ```json
    [
        { "breed": "Siamese", "count": 12 },
        { "breed": "Maine Coon", "count": 5 }
    ]
    ```
2. Пример топа усыновителей
   ```json
    [
        { "id": 1, "firstName": "Ivan", "lastName": "Ivanov", "count": 4 },
        { "id": 2, "firstName": "Anna", "lastName": "Petrova", "count": 2 }
    ]
    ```

## Теоретическая часть для выполнения задания

1. Агрегация данных
   Агрегация — это процесс объединения множества строк в одно итоговое значение.
   * `COUNT`, `SUM`, `AVG`: когда вызывается `COUNT(id)`, база данных не возвращает объекты, она сканирует индексы и возвращает число.
   * `GROUP BY` -  оператор, который критически важен для аналитики. Он заставляет базу данных разделить все записи на «корзины» (например, по породам) и выполнить агрегатную функцию отдельно для каждой корзины.
   * `WHERE` - фильтрует строки до того, как они будут сгруппированы
   * `HAVING` — фильтрует строки после агрегации (например, найти породы, которых в приюте больше 5 особей).
2. QueryBuilder
   Методы стандартного репозитория (`find`, `findOne`) генерируют SQL автоматически, скрывая детали. QueryBuilder дает полный контроль над запросом:
   * При использовании `getMany()` TypeORM превращает строки таблицы в экземпляры класса `CatEntity` (это называется маппинг).
   * Если считается количество кошек или средний возраст, в сущности `CatEntity` нет таких полей. В этом случае используется `getRawMany()`, который возвращает простой массив объектов JS (Plain Old JavaScript Objects).
   * При написании запроса таблице выдается короткое имя: `.createQueryBuilder('cat')`. Это позволяет избежать конфликтов имен при объединении таблиц (Joins).
3. Реляционные связи и стратегии JOIN
   В этом задании происходит работа со связью `Many-to-One` (Много кошек -> Один владелец). Чтобы получить данные из обеих таблиц за один запрос, используются `Join`-ы:
   * `Inner Join (innerJoin)`: Представьте два круга (Кошки и Люди). Inner Join — это область их пересечения. Если у кошки нет владельца, она исчезнет из результатов запроса. Именно поэтому этот тип подходит для отчета «Топ усыновителей» (нам не нужны люди без кошек и кошки без людей).
   * `Left Join (leftJoinAndSelect)`: Возвращает все записи из «левой» таблицы (Кошки), даже если в «правой» (Люди) пусто. В этом случае поля владельца будут заполнены null. Это идеальный выбор для общего списка животных, где мы хотим видеть всех, вне зависимости от их статуса.


## 🛠 Технологический стек

* **Runtime:** Node.js (v18+)
* **Framework:** NestJS
* **Database:** PostgreSQL (в Docker-контейнере)
* **ORM:** TypeORM
* **API Documentation:** Swagger UI
* **Validation:** Class-validator

## Инструкция по запуску

### 1. Первый запуск (Настройка)

Выполняется один раз при установке проекта.

1.  **Установка зависимостей:**
    ```bash
    npm install
    ```
2.  **Запуск базы данных (Docker):**
    Убедитесь, что Docker Desktop запущен, затем выполните:
    ```bash
    docker-compose up -d
    ```
    *Система создаст контейнер с PostgreSQL на порту `5433`.*

3.  **Запуск приложения:**
    ```bash
    npm run start:dev
    ```
    При возникновении ошибок на данном шаге типа:
    `EADDRINUSE: address already in use :::3000` необходимо сменить порт в самом коде или остановить процессы, которые занимают данный порт, так как ошибка означает, что порт 3000 уже занят. Освободить порт можно при помощи команды:
    ```bash
    Stop-Process -Id (Get-NetTCPConnection -LocalPort 3000).OwningProcess -Force
    ```
    После этого снова выполните запуск приложения `npm run start:dev`

### 2. Последующие запуски

Для повседневной работы достаточно убедиться, что контейнер БД запущен, и запустить сервер:
1. `docker-compose start` (если контейнер был остановлен)
2. `npm run start:dev`

Приложение будет доступно по адресу: [http://localhost:3000/api](http://localhost:3000/api)
