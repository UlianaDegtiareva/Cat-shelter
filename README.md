# 🐾 Система управления приютом для кошек (Cat Shelter API)

## Постановка задачи №4: Безопасность и идентификация (JWT Auth)

**Цель**: : Обеспечить сохранность данных и разграничить права доступа. В приюте накопилось много ценной информации, и теперь операции, влияющие на состояние базы данных (регистрация животных, редактирование профилей, оформление усыновления), должны быть доступны только зарегистрированным сотрудникам (волонтерам). Необходимо внедрить систему аутентификации на базе современных стандартов безопасности.

**Основная задача** - реализовать систему регистрации и входа, а также защитить существующие эндпоинты управления данными.

### Модуль авторизации (`AuthModule`)

| Метод | Путь | Описание | Бизнес-логика |
| :--- | :--- | :--- | :--- |
| **POST** | `/auth/register` | Регистрация волонтера | Создание новой учетной записи. Пароль должен храниться в зашифрованном виде. Логин должен быть уникальным. |
| **POST** | `/auth/login` | Вход в систему | Проверка учетных данных. В случае успеха выдается JWT-токен, который служит «цифровым пропуском» для защищенных действий. |

### Правила доступа:

После внедрения системы авторизации, доступ к API разделяется на две зоны:
1. Публичная зона (Public): Доступна всем без исключения.
   * Просмотр списка кошек (`GET /cats`).
   * Просмотр деталей конкретной кошки (`GET /cats/:id`).
   * Регистрация нового аккаунта (`POST /auth/register`).
2. Защищенная зона (`Protected`): Требует наличия валидного JWT-токена в заголовке `Authorization: Bearer <token>`.
   * Все операции создания (`POST`), обновления (`PATCH`) и удаления (`DELETE`) в модулях `Cats`, `Users` и `Stats`.
   * Попытка доступа к этим методам без пропуска должна приводить к отказу.

### Ожидаемые форматы ответов (API Contract)

Ниже представлены статусы ответов, которые система должна возвращать при обращении к эндпоинтам. Убедитесь, что описания в коде и декораторах @ApiResponse совпадают с указанными ниже.
**Модуль авторизации (Auth Module):**
1. Регистрация волонтера (`POST /auth/register`)
   * **201 Created**: "User successfully registered and token returned."
   * **400 Bad Request**: "Validation error (e.g. short password or empty fields)."
   * **409 Conflict**: "A user with this login already exists."
2. Вход в систему (`POST /auth/login`)
   * **200 OK**: "Returns JWT token."
   * **400 Bad Request**: "Invalid input format or missing fields."
   * **401 Unauthorized**: "Invalid login or password."

Для всех методов, которые требуют авторизации перед выполнением необходимо прописать:
`@ApiResponse({ status: 401, description: 'Not authorized: No token provided or token invalid.' })`

## Теоретическая часть для выполнения задания

1. **Хеширование паролей**
   Хранить пароли в открытом виде (в виде обычного текста) в базе данных — критическая ошибка безопасности.
   * `Bcrypt`: Алгоритм, который превращает пароль в «хеш». Даже если злоумышленник получит доступ к базе, он не сможет узнать исходный пароль.
   * Соль (`Salt`): Случайная строка, добавляемая к паролю перед хешированием, чтобы защититься от таблиц предвычисленных хешей.
2. **JWT (JSON Web Token)**
   Это компактный формат для безопасной передачи информации между сторонами.
   * `Header`: Тип токена и алгоритм подписи.
   * `Payload`: Данные пользователя (например, id и username), которые мы хотим «зашить» в токен.
   * `Signature`: Гарантирует, что токен не был изменен в пути. Проверяется с помощью секретного ключа (Secret Key), известного только серверу.
3. **Guards (Стражи)**
   В NestJS это классы, которые определяют, будет ли запрос обработан контроллером или отклонен.
   * `JwtAuthGuard` автоматически проверяет наличие и валидность токена в заголовке запроса. Если токен валиден, он извлекает данные пользователя и прикрепляет их к объекту запроса (request.user).

## Инструкция по запуску

### 1. Первый запуск (Настройка)

Выполняется один раз при установке проекта.

1.  **Установка зависимостей:**
    ```bash
    npm install
    ```
2.  **Запуск базы данных (Docker):**
    Убедитесь, что Docker Desktop запущен, затем выполните:
    ```bash
    docker-compose up -d
    ```
    *Система создаст контейнер с PostgreSQL на порту `5433`.*

3.  **Запуск приложения:**
    ```bash
    npm run start:dev
    ```
    При возникновении ошибок на данном шаге типа:
    `EADDRINUSE: address already in use :::3000` необходимо сменить порт в самом коде или остановить процессы, которые занимают данный порт, так как ошибка означает, что порт 3000 уже занят. Освободить порт можно при помощи команды:
    ```bash
    Stop-Process -Id (Get-NetTCPConnection -LocalPort 3000).OwningProcess -Force
    ```
    После этого снова выполните запуск приложения `npm run start:dev`

### 2. Последующие запуски

Для повседневной работы достаточно убедиться, что контейнер БД запущен, и запустить сервер:
1. `docker-compose start` (если контейнер был остановлен)
2. `npm run start:dev`

Приложение будет доступно по адресу: [http://localhost:3000/api](http://localhost:3000/api)
