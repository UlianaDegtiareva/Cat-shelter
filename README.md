# 🐾 Система управления приютом для кошек (Cat Shelter API)

## Постановка задачи №6: Внешние интеграции

**Цель**: Реализовать механизм взаимодействия с внешней государственной информационной системой «РосКотМониторинг». Теперь система приюта должна не только вести внутренний учет, но и обеспечивать официальную регистрацию животных (чипирование) через внешний API, обеспечивая надежность обмена данными и обработку внештатных ситуаций на стороне внешнего сервиса.

**Основные задачи**:

1. Внедрить HttpModule (на базе Axios) для выполнения запросов к сторонним API.
2. Реализовать механизм безопасной авторизации во внешней системе с использованием секретных ключей (API Keys), хранящихся в переменных окружения (.env).
3. Создать защищенный эндпоинт для регистрации чипа, доступный только администраторам системы.
4. Разработать Mock-сервис (имитатор) для тестирования интеграции в условиях отсутствия доступа к реальной государственной базе.

### Модуль интеграции с РосКотМониторингом

| Метод | Путь | Описание | Правила доступа (Business Logic) |
| :--- | :--- | :--- | :--- |
| **POST** | `/cats/:id/chip` | Регистрация чипа | Только Admin. Берет данные кошки из БД и отправляет во внешнюю систему. Полученный chipId сохраняется в профиль кошки. |
| **POST** | `/external-api/roskot/register-chip` | Имитатор API | Внутренний Mock. Принимает JSON с именем и породой, возвращает уникальный номер чипа в формате RU-STATE-XXXXX. |

### Ожидаемые форматы ответов (API Contract)

Ниже представлены статусы ответов, которые система должна возвращать при обращении к эндпоинтам. Убедитесь, что описания в коде и декораторах @ApiResponse совпадают с указанными ниже.
**Модуль чипирования (Chipping Module):**

1. Регистрация во внешней системе (`POST /cats/:id/chip`)
   * **201 Created**: "Chipping successful, code received."
   * **400 Bad Request**: "The cat is already microchipped."
   * **404 Not Found**: "Cat not found."
   * **502 Bad Gateway**: "RosKotMonitoring Error: [Details]"

Для всех методов, которые требуют авторизации перед выполнением необходимо прописать:
`@ApiResponse({ status: 401, description: 'Not authorized: No token provided or token invalid.' })`

Для успешной сдачи задания имитатор внешней системы должен поддерживать сценарии "нестабильной работы", которые ваш сервис обязан корректно обрабатывать:

* **Simulated Error**: При передаче имени кошки "SystemError", имитатор возвращает 500 Internal Server Error.
* **Network Timeout**: При передаче имени кошки "Slowy", имитатор задерживает ответ на 10 секунд (ваш сервис должен оборвать соединение по таймауту в 5 сек).
* **Invalid API Key**: При неверном или отсутствующем заголовке x-api-key возвращается 401 Unauthorized.

## Теоретическая часть для выполнения задания

В NestJS для работы с внешними ресурсами используется обертка над Axios. Запросы возвращают Observable, которые необходимо преобразовывать в Promise с помощью функции lastValueFrom для удобной работы через async/await.
Конфиденциальные данные (URL внешнего API и секретные ключи) нельзя хранить в коде. Они выносятся в файл .env и считываются через ConfigService. Для отправки данных во внешнюю систему создается объект-посредник, который содержит только те поля, которые ожидает внешний API (в нашем случае name и breed).

Для успешного выполнения задания ваше решение должно строго соответствовать API-контракту:

1. JSON-спецификация: Вы можете получить свой актуальный контракт в формате JSON по адресу: http://localhost:3000/api-json (при запущенном сервере).
2. Визуализация контракта: Чтобы ознакомиться с интерактивной структурой запросов, моделей данных и статус-кодов:
   * Скопируйте содержимое файла openapi.json (конракт, который должен получиться после выполнения задания) из корневой директории проекта.
   * Перейдите в [онлайн-редактор](https://editor.swagger.io/).
   * Вставьте скопированный текст в левое окно редактора.
   * В правой части экрана отобразится полная визуализация всех эндпоинтов и требований к валидации.

## Инструкция по запуску

### 1. Первый запуск (Настройка)

Выполняется один раз при установке проекта.

1.  **Установка зависимостей:**
    ```bash
    npm install
    ```
2.  **Запуск базы данных (Docker):**
    Убедитесь, что Docker Desktop запущен, затем выполните:
    ```bash
    docker-compose up -d
    ```
    *Система создаст контейнер с PostgreSQL на порту `5433`.*

3.  **Запуск приложения:**
    ```bash
    npm run start:dev
    ```
    При возникновении ошибок на данном шаге типа:
    `EADDRINUSE: address already in use :::3000` необходимо сменить порт в самом коде или остановить процессы, которые занимают данный порт, так как ошибка означает, что порт 3000 уже занят. Освободить порт можно при помощи команды:
    ```bash
    Stop-Process -Id (Get-NetTCPConnection -LocalPort 3000).OwningProcess -Force
    ```
    После этого снова выполните запуск приложения `npm run start:dev`

### 2. Последующие запуски

Для повседневной работы достаточно убедиться, что контейнер БД запущен, и запустить сервер:
1. `docker-compose start` (если контейнер был остановлен)
2. Для этого задания потребуются новые библиотеки, установить которые можно при помощи команды:
   ```bash
      npm install @nestjs/jwt @nestjs/passport bcrypt passport-jwt
   ```
3. `npm run start:dev`

Приложение будет доступно по адресу: [http://localhost:3000/api](http://localhost:3000/api)
