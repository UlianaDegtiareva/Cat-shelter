# 🐾 Система управления приютом для кошек (Cat Shelter API)

## Постановка задачи №6: Высокопроизводительный Reader-сервис на Go

**Цель**: Оптимизировать чтение данных и разгрузить основное приложение NestJS. В этой задаче мы внедряем микросервисную архитектуру, где тяжелые операции чтения (списки кошек, профили пользователей, сложная статистика) делегируются сервису на языке Go, а операции записи и безопасность остаются на стороне NestJS.

**Основные задачи**:

1. Разделение ответственности (CQRS): NestJS теперь выступает в роли API Gateway (прокси-сервер), а Go — в роли быстрого поставщика данных.
2. Реализовать проксирование GET-запросов через proxy-fetcher из NestJS в Go.
3. Написать SQL-запросы в Go для моментального получения данных с использованием JOIN и агрегации статистики.

Теперь NestJS не обращается к своей базе данных для GET-запросов, а перенаправляет их в Go-сервис (порт 8080 внутри Docker). Для того чтобы система выглядела единой, Go-сервис должен возвращать ошибки в формате, который понимает NestJS.

## Теоретическая часть для выполнения задания

1. **Паттерн API Gateway (Proxy)**
   В этой схеме клиент (`Swagger`) общается только с `NestJS`. `NestJS`, получив GET-запрос, выступает в роли "клиента" для Go-сервиса. Он делает внутренний запрос, получает данные и отдает их пользователю. Это позволяет скрыть внутреннюю архитектуру от пользователя и сохранить единую точку входа с авторизацией.
2. **Производительность `Go` (`Golang`)**
   `Go` — компилируемый язык, который работает значительно быстрее `Node.js` на операциях ввода-вывода (I/O). Использование чистого SQL (библиотека `database/sql`) вместо ORM позволяет избежать лишних накладных расходов при маппинге данных.
3. **Обработка NULL в `Go` (`sql.NullString`)**
   В реляционных базах данных поля после LEFT JOIN могут быть пустыми (`NULL`).
   * В `Go` обычная переменная `string` не может принять значение `NULL` (это вызовет ошибку).
   * Мы используем типы `sql.NullString` и `sql.NullInt64`, чтобы безопасно считывать данные и проверять их наличие через поле `.Valid`.

## 🛠 Технологический стек

* **Gateway Framework:** NestJS (Node.js)
* **Reader Engine:** Go 1.21+ (Gin Framework)
* **Database:** PostgreSQL (в Docker-контейнере)
* **ORM:** TypeORM
* **API Documentation:** Swagger UI
* **Validation:** Class-validator

## Инструкция по запуску

### 1. Первый запуск (Настройка)

Выполняется один раз при установке проекта.

1.  **Установка зависимостей:**
    ```bash
    npm install
    ```
2.  **Запуск базы данных (Docker):**
    Убедитесь, что Docker Desktop запущен, затем выполните:
    ```bash
    docker-compose up -d
    ```
    *Система создаст контейнер с PostgreSQL на порту `5433`.*

3.  **Запуск приложения:**
    ```bash
    npm run start:dev
    ```
    При возникновении ошибок на данном шаге типа:
    `EADDRINUSE: address already in use :::3000` необходимо сменить порт в самом коде или остановить процессы, которые занимают данный порт, так как ошибка означает, что порт 3000 уже занят. Освободить порт можно при помощи команды:
    ```bash
    Stop-Process -Id (Get-NetTCPConnection -LocalPort 3000).OwningProcess -Force
    ```
    После этого снова выполните запуск приложения `npm run start:dev`

### 2. Последующие запуски

Для повседневной работы достаточно убедиться, что контейнер БД запущен, и запустить сервер:
1. `docker-compose start` (если контейнер был остановлен)
2. Для этого задания потребуются новые библиотеки, установить которые можно при помощи команды:
   ```bash
      npm install @nestjs/jwt @nestjs/passport bcrypt passport-jwt
   ```
3. `npm run start:dev`

Приложение будет доступно по адресу: [http://localhost:3000/api](http://localhost:3000/api)
