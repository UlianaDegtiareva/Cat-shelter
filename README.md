# 🐾 Система управления приютом для кошек (Cat Shelter API)

## Постановка задачи №2: Реляционные модели и автоматизация бизнес-процессов

**Цель**: Необходимо перейти от плоского списка животных к реляционной структуре данных. Требуется реализовать учет волонтеров (пользователей) и автоматизировать процесс передачи животного из приюта новому владельцу.

### Функциональные требования:

1. Управление пользователями:
   * Создать сущность `User` (поля: `firstName`, `lastName`).
   * Реализовать стандартные операции: создание пользователя и получение списка.
2. Реляционная логика:
   * Реализовать связь «Один ко многим»: один пользователь может быть владельцем нескольких кошек.
   * В сущности кошки предусмотреть поле `adoptionDate` (дата усыновления) и `isAdopted` (флаг статуса).
   * Настроить поведение базы данных при удалении пользователя так, чтобы записи о кошках не удалялись (`SET NULL`).
   * Реализовать возможность фильтрации списка животных по породе (`breed`) и текущему статусу усыновления (`isAdopted`).
3. Реализовать бизнес-логику усыновления:
   * Вход: `ID кошки` и `ID пользователя`.
   * Кошка должна существовать и быть свободна (`isAdopted: false`). Пользователь должен существовать.
   * Привязать кошку к пользователю, установить статус `true` и автоматически зафиксировать текущее время сервера.
  
### Модуль Cats

| Метод  | Путь               | Описание                          | Статус |
| :----- | :----------------- | :-------------------------------- | :----- |
| POST   | `/cats`            | Регистрация новой кошки           | 201    |
| GET    | `/cats`            | Список всех кошек (с фильтрами)   | 200    |
| GET    | `/cats/:id`        | Детальная информация + информация о владельце (если есть)   | 200    |
| PATCH  | `/cats/:id`        | Обновление данных (кроме статуса усыновления) | 200    |
| PATCH  | `/cats/:id/adopt`  | Процесс усыновления (автоматическая установка даты)   | 200    |
| DELETE | `/cats/:id`        | Удаление записи                   | 204    |

### Модуль Users

| Метод  | Путь               | Описание                          | Статус |
| :----- | :----------------- | :-------------------------------- | :----- |
| POST   | `/users`            | Создание волонтера/владельца           | 201    |
| GET    | `/users`            | Список всех пользователей   | 200    |
| GET    | `/users/:id`            | Получение данных конкретного пользователя   | 200    |
| GET    | `/users/:id/cats`        | Список кошек конкретного владельца   | 200    |
| DELETE | `/users/:id`        | Удаление записи (с проверкой SET NULL)                   | 204    |

Ваш контроллер должен строго реализовывать следующие коды состояния и описания ответов для корректной работы автоматических тестов и генерации Swagger:

Модуль кошек (Cats Module):
1. Регистрация новой кошки (`POST /cats`)
   * **201 Created**: "The cat has been successfully registered."
   * **400 Bad Request**: "Invalid input data."
   * **409 Conflict**: "Conflict: A cat with this name already exists."
2. Обновление данных (`PATCH /cats/:id`)
   * **200 OK**: "Cat info updated successfully."
   * **400 Bad Request**: "Invalid input or ID"
   * **404 Not Found**: "Cat not found."
   * **409 Conflict**: "New name already taken"
3. Поиск по базе приюта (`GET /cats`)
   * **200 OK**: "Success"
4. Детальная информация о кошке (`GET /cats/:id`)
   * **200 OK**: "Cat data retrieved successfully."
   * **400 Bad Request**: "Invalid ID format. Expected an integer."
   * **404 Not Found**: "Cat not found."
5. Процесс усыновления (`PATCH /cats/:id/adopt`)
   * **200 OK**: "Adoption process completed."
   * 400 Bad Request: "Cat is already adopted or invalid user ID."
   * 404 Not Found: "Cat or User not found."
6. Удаление записи (`DELETE /cats/:id`)
   * **204 No Conten**t: "Record deleted successfully."
   * **400 Bad Request**: "Invalid ID format. Expected an integer."
   * **404 Not Found**: "Cat not found, cannot delete."

Модуль пользователей (Users Module):
1. Создание волонтера (`POST /users`)
   * **201 Created**: "User successfully created."
   * **400 Bad Request**: "Validation failed"
2. Список всех пользователей (`GET /users`)
   * **200 OK**: "Returns a list of all registered users."
3. Профиль пользователя (`GET /users/:id`)
   * **200 OK**: "User found."
   * **400 Bad Request**: "Invalid ID format provided."
   * **404 Not Found**: "User not found."
4. Список кошек владельца (`GET /users/:id/cats`)
   * **200 OK**: "Success"
   * **400 Bad Request**: "Invalid ID"
   * **404 Not Found**: "User not found"
5. Удаление пользователя (`DELETE /users/:id`)
   * **204 No Content**: "User deleted successfully."
   * **400 Bad Request**: "Invalid ID"
   * **404 Not Found**: "User not found."

Для каждого эндпоинта используйте декораторы `@ApiResponse({ status: X, description: '...' })` или специализированные версии, такие как `@ApiNotFoundResponse()`. Помните, что для успешного удаления (`DELETE`) необходимо также использовать декоратор `@HttpCode(204)`.

Для успешного выполнения задания ваше решение должно строго соответствовать API-контракту:

1. JSON-спецификация: Вы можете получить свой актуальный контракт в формате JSON по адресу: http://localhost:3000/api-json (при запущенном сервере).
2. Визуализация контракта: Чтобы ознакомиться с интерактивной структурой запросов, моделей данных и статус-кодов:
   * Скопируйте содержимое файла openapi.json (конракт, который должен получиться после выполнения задания) из корневой директории проекта.
   * Перейдите в [онлайн-редактор](https://editor.swagger.io/).
   * Вставьте скопированный текст в левое окно редактора.
   * В правой части экрана отобразится полная визуализация всех эндпоинтов и требований к валидации.

### Технические требования:
1. Использовать `class-validator` для проверки имен (минимум 2 символа) и возраста (неотрицательное число).
2. Все эндпоинты должны быть документированы. Ошибки `404 (Not Found)` и `400 (Bad Request)` должны быть описаны.
3. Использовать TypeORM relations для подгрузки вложенных объектов (при получении данных о кошке должен быть виден её владелец).
4. При вызове метода `/adopt` в базе данных обновляются 3 поля: `ownerId`, `isAdopted`, `adoptionDate`.
5. Попытка усыновить уже усыновленную кошку возвращает ошибку `400 Bad Request`.

## Инструкция по запуску

### 1. Первый запуск (Настройка)

Выполняется один раз при установке проекта.

1.  **Установка зависимостей:**
    ```bash
    npm install
    ```
2.  **Запуск базы данных (Docker):**
    Убедитесь, что Docker Desktop запущен, затем выполните:
    ```bash
    docker-compose up -d
    ```
    *Система создаст контейнер с PostgreSQL на порту `5433`.*

3.  **Запуск приложения:**
    ```bash
    npm run start:dev
    ```
    При возникновении ошибок на данном шаге типа:
    `EADDRINUSE: address already in use :::3000` необходимо сменить порт в самом коде или остановить процессы, которые занимают данный порт, так как ошибка означает, что порт 3000 уже занят. Освободить порт можно при помощи команды:
    ```bash
    Stop-Process -Id (Get-NetTCPConnection -LocalPort 3000).OwningProcess -Force
    ```
    После этого снова выполните запуск приложения `npm run start:dev`

### 2. Последующие запуски

Для повседневной работы достаточно убедиться, что контейнер БД запущен, и запустить сервер:
1. `docker-compose start` (если контейнер был остановлен)
2. `npm run start:dev`

Приложение будет доступно по адресу: [http://localhost:3000/api](http://localhost:3000/api)
